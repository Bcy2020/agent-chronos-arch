# 递归分解 MVP

本目录包含一个实验性 MVP，用于在 `agent-chronos-arch` 项目中验证**递归系统分解**方法。

其目标**不是**直接生成可直接上线的架构。  
其目标是测试：一个 PRD 能否被递归分解为一棵函数块树，且满足以下要求：

- 明确的边界
- 明确的输入与输出
- 明确的子节点交互流程
- 明确的接口保持推理

## 为什么需要这个

对于复杂系统，直接让 AI 跳到函数级设计通常是不可靠的。

问题通常不在于 AI 完全无法分解，而在于：

- 大系统上下文会变得嘈杂且不稳定
- 函数级粒度过早变得不一致
- 不同分支之间容易出现职责重复与冲突
- 最终得到的分解结果难以验证

本实验探索另一条路径：  
一个**自顶向下、递归、最小上下文**的分解过程。

## 核心思想

本实验中的分解方法论与设计原则由 **Shining Pau** 提出。  
当前原型代码由 AI 基于这些思想编写。

核心过程如下：

1. 将整个软件系统视为一个顶层的函数块。
2. 要求每个函数块必须定义：
   - 清晰的 `purpose`（目的）
   - 清晰的 `inputs`（输入）
   - 清晰的 `outputs`（输出）
   - 清晰的 `boundary`（边界：`in_scope` / `out_of_scope`）
3. 不让 AI 持续在整个系统上做推理，而是让它只工作于**当前块**，并递归地将当前块分解为子块。
4. 对每个子块施加相同的约束，直到分解达到足够小、足够稳定、可独立实现与测试的单元。
5. 要求分解结果必须说明：
   - 子节点之间的信息如何流动
   - 子节点如何共同实现父节点
   - 子节点组合后的外部接口是否与父节点接口一致

换句话说，本实验不是关于“将需求拆成更小的文本片段”，  
而是关于**将需求递归地转换为一个可组合的契约结构**。

## 最小上下文原则

本实验的一个关键假设是：

> 当 AI 对局部节点进行分解时，它不应携带整个系统的上下文。  
> 它只应接收当前节点的完整边界。

因此，每一次递归步骤的理想输入只包含：

- 当前节点的名称与目的
- 当前节点的输入与输出
- 当前节点的边界
- 当前节点的父契约

这样设计的目的是：减少无关上下文，提高局部分解的稳定性。

## 接口保持原则

该方法的一个核心要求是：

> 父节点被分解为子节点之后，由这些子节点组合而成的系统，必须对外暴露与父节点相同的外部接口。

这意味着：

- 父节点的输入必须被子节点消费
- 父节点的输出必须由子节点产出
- 额外的中间产物必须保持内部可见
- 正确性不能依赖隐含或未声明的协调者

这也是本实验与普通任务拆解的主要区别之一。

## 当前 MVP 的能力

当前 MVP 支持：

- 接收 PRD 作为输入
- 使用真实 LLM 或 Stub 后端进行递归分解
- 为每个节点生成：
  - `purpose`
  - `inputs`
  - `outputs`
  - `boundary`
  - `child_interaction_flow`
  - `coverage_explanation`
  - `interface_preservation_proof`
- 执行轻量级本地校验，以发现类似以下问题：
  - 泄露了外部接口
  - 父节点的输入未被覆盖
  - 父节点的输出缺失
  - 组合结构不完整

## 当前状态

这是一个**方法论验证原型**，而不是一个生产就绪的系统。

其价值在于验证这种分解范式是否可行，而非替代正式的软件设计流程。

当前已知的局限性包括：

- 停止条件在部分情况下仍依赖递归深度
- 隐含的协调者可能仍然出现在某些输出中
- 接口保持的证明并不总是完全闭合
- 重复职责与共享依赖尚未被完整建模

## 研究问题

本实验不是要回答：

> “AI 能否生成一棵分解树？”

而是要回答：

> **在最小上下文的约束下，AI 能否将系统递归分解为一棵带有明确边界、组合推理和接口保持证据的函数块树？**
