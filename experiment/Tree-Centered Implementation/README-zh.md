# 树中心实现

一个针对**树中心实现优化**架构的概念验证实现，通过递归分解实现自动化代码生成。

## 概述

本 MVP 验证了基于分解的代码生成方法的核心概念：

1. **语义停止条件** — 分解在节点成为纯函数或原子操作时停止，而非基于任意行数
2. **全局状态守恒** — 父节点将所有数据操作委托给子节点，确保状态变更可追溯
3. **分解-验证循环** — 每次分解在继续之前都会通过组合进行验证

## 当前问题

### 高优先级

| 问题 | 影响 | 描述 |
|------|------|------|
| **全局状态重置** | 运行时失败 | `tasks` 和 `next_id` 在每次函数调用时被重新初始化，导致调用间数据丢失 |
| **next_id 更新丢失** | 功能性缺陷 | `generate_task_id()` 返回更新后的 `next_id`，但未持久化回调用方 |
| **缺少导入声明** | 运行时失败 | 生成的代码中使用了 `datetime` 和 typing 导入，但未声明 |

### 中优先级

| 问题 | 影响 | 描述 |
|------|------|------|
| **函数名冲突** | 潜在覆盖 | 不同分支中出现相同的函数名（例如 `complete` 和 `delete` 处理器中都有 `validate_task_exists`） |
| **类型注解** | 不完整 | 部分生成代码使用了类型，但缺少相应的导入 |
| **状态管理** | 架构缺口 | 整个应用缺少统一的状态管理 |

### 低优先级

| 问题 | 影响 | 描述 |
|------|------|------|
| **未执行测试** | 验证缺口 | 代码仅通过 AST 验证，未实际执行 |
| **缺少错误处理** | 健壮性 | 生成的代码缺少全面的错误处理 |
| **缺少横切关注点** | 架构缺口 | 日志、缓存、认证等问题未被处理 |

## 2.0 路线图

### 阶段 1：自动组装模块

**目标**：自动将生成的节点文件组合成可运行的应用程序

- [ ] 导入管理 — 收集并去重所有需要的导入
- [ ] 状态管理器类 — 用 `TaskStore` 类替换模块级全局变量
- [ ] 函数命名空间解析 — 解决命名冲突
- [ ] 入口点生成 — 创建主 CLI 或 API 入口

### 阶段 2：执行测试

**目标**：不仅验证语法，还验证运行时行为

- [ ] 为每个叶子节点生成单元测试
- [ ] 为每个父节点生成集成测试
- [ ] 为数据源集成 Mock 框架
- [ ] 测试执行与反馈循环

### 阶段 3：增强验证

**目标**：在分解阶段发现更多问题

- [ ] 全局状态守恒验证
- [ ] 数据操作类型检查
- [ ] 父子节点之间的契约兼容性验证
- [ ] 数据依赖中的循环检测

### 阶段 4：横切关注点

**目标**：处理非功能性需求

- [ ] 日志注入
- [ ] 错误处理模式
- [ ] 外部服务的重试机制
- [ ] 基础缓存策略

## 架构
PRD 输入 → 分解器 (LLM) → 节点树 → 代码生成器 (LLM) → Python 文件  
↑ ↓  
└── 验证 ← AST 检查 + 子节点使用检查  


### 核心组件

| 文件 | 用途 |
|------|------|
| `main.py` | CLI 入口，包含配置选项 |
| `decomposer.py` | 基于 LLM 的节点分解，支持语义停止条件 |
| `code_generator.py` | 基于 LLM 的代码生成，用于父节点和叶子节点 |
| `validator.py` | AST 验证和子函数使用验证 |
| `models.py` | 节点、契约和数据源的数据结构 |
| `tree_builder.py` | 递归树构建，采用深度优先遍历 |
