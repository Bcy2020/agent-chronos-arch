# 树中心实现改进方案

> **作者说明**
>
> 本文档描述的改进方案由仓库作者提出。
> 文档本身由AI辅助撰写。

## 概述

本文档提出对原有架构的改进方案：使**分解树**成为实现、验证和后续维护的核心结构。

该方法不再区分独立的工作流设计阶段和函数开发阶段，而是让系统在节点被分解后**立即通过子节点接口实现每个节点**。

目标是使整个过程更简单、更灵活、更易于本地验证。

## 核心思想

当一个节点被分解为子节点后，一个面向代码的LLM被要求在一条规则下实现**父节点**：

> 父节点必须通过组合子节点函数来实现。

此时，子节点可能尚未完全实现，但它们的接口已经定义好了。

这提供了一种早期的结构性检查：

- 如果父节点无法通过子节点组合实现，分解可能存在缺陷。
- 如果子节点接口缺失或形状不佳，应修订契约。
- 如果父节点可以通过子节点调用干净地实现，分解就获得了实践支持。

从这个意义上说，组合成为了一种验证机制。

## 语义停止条件

分解不应依赖简单的启发式方法（如行数估计）。相反，停止条件应由节点的**语义分析**决定。

### 节点类型分类

每个节点可分为四类之一：

| 节点类型 | 描述 | 是否停止 |
|---------|------|----------|
| **纯计算** | 无副作用的数学函数。相同输入必定产生相同输出。 | 是 |
| **原子操作** | 对单个数据源的单一操作类型的直接操作。 | 是 |
| **协调节点** | 编排多个子节点，包含业务逻辑、分支或循环。 | 否 |
| **混合节点** | 同时包含计算和协调。 | 否 |

### 停止条件规则

节点应在以下情况停止分解：

1. **纯函数测试**：节点仅执行数学变换，且没有：
   - 全局变量依赖
   - I/O操作
   - 状态修改
   - 外部API调用

2. **原子操作测试**：节点仅对一个数据源执行一种操作：
   - 从单个数据源读取
   - 向单个数据源写入
   - 对单个数据源的读-改-写

3. **达到最大深度**：树已达到配置的最大深度（强制停止）。

如果以上条件都不满足，节点必须继续分解。

### 示例

```
process_order → 协调节点 → 继续分解
├── validate_request → 纯计算 → 停止
├── reserve_inventory → 原子操作 (inventory: 写) → 停止
├── calculate_totals → 纯计算 → 停止
└── send_notification → 原子操作 (notification_service: 写) → 停止
```

## 全局状态前置定义

在分解开始之前，必须明确识别和定义所有数据源。这将**隐式依赖**转变为**显式声明**。

### 数据源分类

| 类别 | 示例 | 访问模式 |
|------|------|----------|
| **数据库** | 订单、库存、客户 | CRUD操作、事务 |
| **用户输入** | API请求、CLI参数、表单 | 处理过程中只读 |
| **文件系统** | 配置文件、日志、模板 | 带I/O开销的读写 |
| **外部服务** | 支付网关、物流API | 异步/同步调用、网络故障 |
| **缓存** | 会话数据、计算结果 | 快速读写、TTL考虑 |

### 前置定义阶段

在分解开始之前：

1. 分析PRD以识别所有数据实体
2. 将每个实体按数据源类别分类
3. 为每个数据源定义访问类型：
   - **只读**：参考数据、配置
   - **读写**：订单、库存、用户资料
   - **只写**：日志、审计追踪、通知
4. 生成指导分解的**数据源清单**

### 数据源清单示例

```yaml
data_sources:
  - name: inventory_database
    category: database
    access: read_write
    entities: [products, stock_levels, reservations]
    operations:
      - reserve_stock(product_id, quantity)
      - release_stock(product_id, quantity)
      - check_availability(product_id)

  - name: order_database
    category: database
    access: read_write
    entities: [orders, order_items]
    operations:
      - create_order(order_data)
      - update_order(order_id, changes)
      - get_order(order_id)

  - name: payment_service
    category: external_service
    access: write_only
    operations:
      - process_payment(amount, method)
      - refund_payment(transaction_id)
```

## 全局状态守恒

一个基本原则：**父节点不得直接操作全局状态**。所有全局操作必须委托给子节点。这称为**全局状态守恒**。

### 守恒定律

```
父节点全局操作 = Σ(子节点全局操作)
```

当父节点有全局操作时，它们必须完全分配给子节点。在分解过程中，任何全局操作都不能"消失"。

### 操作分解规则

| 父节点操作 | 子节点分配方式 | 约束 |
|-----------|---------------|------|
| `read(data_source)` | 一个或多个 `read(data_source)` | 可拆分到不同子节点 |
| `write(data_source)` | 一个或多个 `write(data_source)` | 可拆分到不同子节点 |
| `read_write(data_source)` | 同源的 `read` + `write` | 必须显式分解 |

### 守恒验证

在分解过程中，系统验证：

1. **完整性**：所有父节点全局操作都分配给了子节点
2. **正确性**：子节点仅操作声明的数据源
3. **原子性**：每个子节点每次操作最多操作一个数据源

如果违反守恒，必须修订分解。

### 示例：守恒实践

```
父节点: process_order
├── 全局操作: {inventory: 读写, orders: 读写}
├── 子节点A: validate_items → {inventory: 读}
├── 子节点B: reserve_inventory → {inventory: 写}
├── 子节点C: create_order → {orders: 写}
└── 子节点D: send_confirmation → {} (无全局操作)

验证:
- inventory.读写 = inventory.读 + inventory.写 ✓
- orders.读写 = orders.写 (子节点C通过写操作隐式读取) ✓
```

## 简化工作流

1. **前置定义阶段**
   - 分析PRD中的数据源
   - 生成数据源清单
   - 定义访问类型和操作

2. **分解阶段**
   - 递归构建分解树
   - 为每个节点分配全局操作
   - 验证全局状态守恒
   - 应用语义停止条件

3. **实现阶段**
   - 为子节点定义接口和契约
   - 通过组合子节点调用实现每个父节点
   - 递归继续直到到达叶节点

4. **验证阶段**
   - 自底向上在树中运行测试
   - 验证组合正确性
   - 如果节点失败，返回其原始上下文并修订

## 主要优势

### 分解的早期验证

系统不必等到最终集成才发现分解不可用。

### 语义停止条件

分解不是停在任意阈值，而是停在节点达到其语义本质时：纯函数或原子操作。

### 显式全局状态管理

通过前置定义数据源和强制守恒，系统消除了隐藏依赖并确保可追溯的状态变更。

### 更简单的架构

分解树成为主要的执行结构，减少了过程开销。

### 本地化修复

缺陷可以追溯到特定节点并在原始本地上下文中修复。

### 更好地处理需求变更

需求变更可以通过树结构和接口边界更精确地映射到受影响的节点。

## 重要警告

不应将这种改进理解为仅依赖函数签名。

实际上，每个节点通常需要一个更强的契约，包括以下内容的某种组合：

- 行为期望
- 前置条件和后置条件
- 错误语义
- 状态边界
- 组合约束
- **全局操作声明**

否则，父节点可能在语法层面看起来可组合，但在语义层面仍然是错误的。

## 局限性

分解树是责任分解的强大骨架，但实际系统也包含横切关注点，如日志、认证、缓存和共享基础设施。

因此，树应被视为主要的结构模型，但并非总是完整的依赖模型。

额外考虑：

- **并发**：当前模型假设单线程执行。分布式系统需要额外的协调协议。
- **事务**：跨越多个数据源的多节点操作需要事务管理，不在当前模型范围内。
- **横切关注点**：这些应作为切面或中间件实现，而不是作为树结构的一部分。

## 总结

这种改进将架构从：

- **阶段中心开发**

转变为：

- **树中心实现与验证**

主要主张：

> 分解的判断不仅要看它是否看起来合理，还要看父节点是否能真正通过子节点接口的组合来实现。

> 分解不应停在任意阈值，而应仅当节点达到其语义本质时才停止：纯计算或原子操作。

> 全局状态操作必须在整个分解层次结构中显式声明和守恒。

这使得分解树不仅成为设计产物，还成为实现、测试和变更定位的主要结构。
