# MVP Tier Standard

As the current repository's MVP has evolved from validating tree decomposition toward a full architecture, it is necessary to separate MVPs and manage them with versioning. This document defines the following tier system.

## 1. MVP-0.x Versions

These MVP versions are designed to prove the feasibility of tree decomposition. The workflow takes a PRD document as input and outputs a decomposition tree with corresponding code for each node. The code generated by these MVP versions is not yet connected by a code pipeline and therefore cannot be executed. However, a basic re-decomposition mechanism is already in place.

## 2. MVP-1.x Versions

These MVP versions will introduce a code pipeline, making the generated code executable. A dedicated benchmark for executable code will be established to evaluate this and subsequent MVP versions. However, these MVP versions are limited to generating runnable code — they have not yet introduced error feedback, reverse validation, or other mechanisms described in the architecture documents. As a result, the generated code is still far from真正的 software.

**Later versions will be planned once MVP-1.x is relatively mature.**

*Tip: MVP versions do not follow a strict chronological order. Sometimes, when a new mechanism is needed, it may be introduced into a basic MVP first, tested through iterations, and then incorporated into the latest MVP. (For example, the feedback mechanism may be introduced into an MVP-0.x version before being added to an MVP-1.x version.)*

### Appendix: MVP Evolution Table

| MVP Version | Evolution Description |
|-------------|----------------------|
| MVP-0.1 | The most basic MVP, implemented in Chinese, with only fundamental tree decomposition and re-decomposition capabilities |
| MVP-0.2.0 | Introduced **Json PRD** and **SubPRD** to address significant PRD information loss and distortion during transmission across tree levels |
| MVP-0.2.1 | Enabled LLM JSON formatted output mode based on the previous version |
| MVP-0.3.1 | Targeted the issue where the LLM often repeated the same mistakes during re-decomposition, by implementing LLM retry context feedback,but there are still omissions regarding the information flow at leaf nodes.|
| MVP-0.4.1 | Introduced **Interface Layer** to address leaf nodes misusing operation IDs (`op_root_*`) as Python variables. Added pre-decomposition Interface Planning phase: `ResourceSpec`, `InterfaceSpec`, and CRUD-only operation whitelist (`get/list/create/update/delete/exists`). Data models and LLM planner in place; tree decomposition does not yet consume the InterfacePlan.|
| MVP-0.4.2 | Extended Interface Layer with **Interface CodeGen (Phase 3)** and **Capability Allocation (Phase 4)**. `InterfaceCodeGenerator` generates `interfaces.py` from InterfacePlan with storage-model-specific templates (dict/list CRUD) and `ast.parse` validation. `CapabilityAllocator` matches leaf `requested_capabilities` against InterfacePlan, granting or failing at allocation time. `CodeGenerator` uses a new interface-based leaf prompt that forbids `global`, `op_id`, and `source_id` when granted capabilities are present.|
